Fields
 Standard fields

The new data types all have a standard set of fields to help manage the data. In the data type description, these fields are not completely documented but a link is provided to this page for more information.
For compatibility reasons, the old data types currently doesn't have all these fields; they will be completed when they are completely migrated to the web service.

The following table lists the standard fields.

Name Type Description
DeletedDT DateTime null if the data is valid.
Date and time of the deletion when the data is deleted.
LastChangeDT DateTime Date and time of the last change of the data.
LastChangeUser Int32 Number of the user who makes the last changed of the data.
LastChangeUsername String Name of the user who makes the last changed of the data.
No Int32 Unique number to identify the data.
Version Int32 Version of the data.
This value is always incremented when a data is changed in a table, but globally.

When you want a full list of data, the request has no Version attribute on it and you will receive the full list of not-deleted entries. When you want an update to a list, you specify the Version attribute with the highest version you have in your current list and you will receive only the data that have a version higher than the parameter, but including those with a DeletedDT value set. To update your in-memory list, you have just to:

  • Delete the entries that have a DeletedDT value.
  • Update the entries that have no DeletedDT value, but that you have in your list (search them using the No field).
  • Insert the entries that have no DeletedDT value, but you cannot find in your current list.
  • Update the version of your list with the received entries.

Note that, to be sure to have all the changes, you must not specify a filter when you request changes to a list, because a change in a data can also mean that a data can appear in your list or can also disappear. So you must filter yourself the updated and inserted entries.

 Accessible fields and NULL values

When you request the data for a field, you can have three different values for the field:

Attribute Signification
Attribute not present The user doesn't have access to the field or the field doesn't exist, so the value is not present.
Attribute present but is empty Indicates that the field contains a NULL value. So there is no defined value.
For a string, it indicates an empty value.
Attribute present with a value The value of the field.
 List of fields

All requests that ask for a list of data must contain a list of the fields that must be sent back. This feature is used to diminish the size of the response from the server. If no field list is specified, the server will use only the default fields for the specified data type.

For requests that ask for a single data, they can contain a list of fields to send back. If no field list is specified, the server will send all the fields the user has access to.

If the field list contains an invalid field, it will be automatically removed and no error will be sent back.

In any case, the server will filter the list with the fields that are accessible to the user specified in the request. If a field is not accessible, it will be removed from the fields to send and no error will be sent back.

The field list is an attribute of the request and has the following format:

List of fields
Copy Code 
Fields="Name1 Name2 ..."

Where Name1, Name2, ... are the name of the fields. The names must be separated by at least a space.

The following format is also accepted by the web service, but it is now deprecated, because it takes more bandwidth to transfer and also takes more time to code and decode:

List of fields
Copy Code 
<Fields>
  <Field Name="Name1" />
  <Field Name="Name2" />
  ...
</Fields>

See the description of the different data formats for a list of valid field names.
