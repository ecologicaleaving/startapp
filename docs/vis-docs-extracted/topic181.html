Request to check a user
 Request

Username and password

Web service request
Copy Code 
<Request Type="CheckUser"
         CheckOldDatabase="<false or true>"
         Password="<password>"
         Username="<username>" />

Where:

Attribute Description
CheckOldDatabase A Boolean value that indicates whether we must check the old database information for the user, instead of the new one.
If this attribute is not specified, its default value is false.
Password The password, either in plain text or encrypted, if the user has one.
Username Username of the user.
This is a mandatory parameter.

UserType, UserTypeNo and Password

When you know the user type, the number associated with the user type and the password of the user, you can use the following format of the request:

Web service request
Copy Code 
<Request Type="CheckUser"
         CheckOldDatabase="<false or true>"
         Password="<password>"
         UserType="<type of user>"
         UserTypeNo="<number for type of user>" />

Where:

Attribute Description
CheckOldDatabase A Boolean value that indicates whether we must check the old database information for the user, instead of the new one.
If this attribute is not specified, its default value is false.
Password The password, either in plain text or encrypted, if the user has one.
UserType The type of the user.
This is a mandatory parameter.
UserTypeNo The number associated to the type of the user.
For example, if the user type is Player, this attribute contains the number of the player; if the user type is VolleyClub, this attribute contains the number of the volleyball club.
This is a mandatory parameter.
 Response

When the request is successful, the service will return one of the following XML elements:

XML element Description
<Anonymous> The specified user exists, but is anonymous and can not log in.
<BadUser> No user has been found using the specified parameters.
<Disabled> The specified user exists, but is disabled.
<OK> The specified user exists and can log in.
<Revoked> The specified user exists, but is revoked and can not log in.
The content of the XML element contains the reason why the user is revoked.
<UserDataError> The specified user exists, but there are errors in its data on the SQL table and he/she can therefore not log in.
The content of the XML element contains the type of data that contains an error.
 Errors

In addition to global errors, the following errors can be sent by the server:

Error Description
<BadParameter> The CheckOldDatabase attribute, if specified, is not a valid Boolean value.
The UserType attribute, if specified, has not a valid value.
The UserTypeNo attribute, if specified, is not a valid 32-bits integer number.
<ParameterMissing> You must specify one whole group of attributes.
 Security

This request is public and can be used by any user.

The password can be encrypted or in plain text.
If you're writing a .NET Windows application, you can use an assembly written by the FIVB to encrypt the password, or even to use a ready-to-use request.

 Example - Username and Password

Check if the Guest user is valid.

Web service request
Copy Code 
<Request Type="CheckUser"
         Username="Guest" />

The response is:

Web service response
Copy Code 
<Anonymous />
