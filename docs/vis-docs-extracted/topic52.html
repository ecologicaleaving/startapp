Volleyball Match

Represents a match in a volleyball tournament.

The name of the XML element for a player is <VolleyballMatch>.
The name of the XML element for a list of players is <VolleyballMatches>.

 Date and time

The are multiple properties to manage the date and time of a match. Here are more information about them.

DateLocal and TimeLocal: the respectively scheduled local date and local time for the match. The DateLocal property has a value when the date is known. The TimeLocal property has a value when the time is known. Contrary to the DateTimeLocal property, you can have the date and not the time using these property. To known whether the date or time must be confirmed, you must use the ScheduleInfo property.

DateTimeLocal: the scheduled local date and time of the match. Has a value when the date and time are known. Sometimes, it is possible to have only the date, or the date must be confirmed, or the time must be confirmed. You must use the ScheduleInfo property to know what is available and confirmed. You should use the DateLocal and TimeLocal properties to display the date and time, because there can be a date and not always a time.

DateTimeUtc: the UTC scheduled date and time of the match. Has a value only when the local date and time are confirmed.

BeginDateTimeUtc: the UTC date and time when the match really begins. Was introduced later than the other properties, so the values are only for new matches and when the information is available.

EndDateTimeUtc: the UTC date and time when the match ends.

 Referees

Initially, the referees were known only by their name and federation code, using the RefereeXFederationCode and RefereeXName properties. Later (starting in 2013), the referees were managed by the VIS system, so the NoRefereeX properties were added.

To have the name and federation of the referees, you can always use the RefereeXFederationCode and RefereeXName properties, which are always available when the referees are known. When the NoRefereeX properties are set, you can use the referees API (currently not documented) to get more information about the referees.

 Sets

A match can be played in 5 or 7 sets. You can see the format of a match with the Format property. When a match is played in 5 sets, it ends as soon as a team has won 3 sets. So the match can have 3, 4 or 5 sets. When a match is played in 7 sets, it ends as soon as a team has won 4 sets. So the match can have 4, 5, 6 or 7 sets.

The durations of the sets (DurationSetX properties) and the set points for each set (PointsTeamASetX and PointsTeamBSetX properties) don't have a value for the sets that are not played. You should take this into account when displayed the result or the duration.

 Teams

There are multiple properties and relations about the teams. Here are more information about them.

NoTeamA and NoTeamB: the number of the registration of the teams. Are set as soon as the teams are known. When the teams are known, you can have more information about them with the use of the TeamA and TeamB relations.

TeamCodeA and TeamCodeB: the codes of the teams. Are set as soon as the teams are known.

TeamNameA and TeamNameB: the names of the teams. Are set as soon as the teams are known.

TeamAText and TeamBText: the text to display when the teams are unknown. Currently this text is in English and indicates, for example, "RUS / KOR" or "A3 or B3".

To simplify the use of theses properties for a client, the following properties are defined:

  • TeamACalculatedCode and TeamBCalculatedCode: is the first defined value from TeamCode and TeamText, in order.
  • TeamACalculatedName and TeamBCalculatedName: is the first defined value from TeamName, TeamCode and TeamText, in order.

Client code should only use these properties.
