Request to send the login information to a user
 Request

By user number

Web service request
Copy Code 
<Request Type="SendUserLoginInfo"
         No="101" />

Where:

Attribute Description
No Number of the user.

By username

Web service request
Copy Code 
<Request Type="SendUserLoginInfo"
         Username="Test-Player" />

Where:

Attribute Description
Username Username of the user.

By e-mail address

Web service request
Copy Code 
<Request Type="SendUserLoginInfo"
         EMailAddress="vis.sdk@fivb.org" />

Where:

Attribute Description
EMailAddress E-mail address of the user.

Note that if there is more than one user with the specified e-mail address, one e-mail will be sent for each user.

By user type and number

Web service request
Copy Code 
<Request Type="SendUserLoginInfo"
         UserType="Player"
         UserTypeNo="101" />

Where:

Attribute Description
UserType Type of the user.
This is a mandatory parameter if UserTypeNo is specified.
UserTypeNo The number associated to the type of the user.
For example, if the user type is Player, this attribute contains the number of the player; if the user type is VolleyClub, this attribute contains the number of the volleyball club.
This is a mandatory parameter if UserType is specified.

By user type and number in the old database

Web service request
Copy Code 
<Request Type="SendUserLoginInfo"
         CheckAlsoOldDatabase="true"
         UserType="Player"
         UserTypeNo="101" />

Where:

Attribute Description
CheckAlsoOldDatabase true if the user must also be checked in the old database.
false if the user must only be checked in the new database.
If this attribute is not specified, its default value is false.
If this attribute is true, the check on the old database will be performed only if no matching user is found in the new database.
For a search in the old database, only a UserType of Player is accepted.
UserType Type of the user.
This is a mandatory parameter if UserTypeNo is specified.
UserTypeNo The number associated to the type of the user.
For example, if the user type is Player, this attribute contains the number of the player; if the user type is VolleyClub, this attribute contains the number of the volleyball club.
This is a mandatory parameter if UserType is specified.
 Response
If the request is successful, the service will return an <OK> XML element.
 Errors

In addition to global errors, the following errors can be sent by the server:

Error Description
<BadEMailAddress> The e-mail address of the user is invalid.
The content of the element is the bad e-mail address.
<BadParameter> The CheckAlsoOldDatabase attribute, if specified, is not a valid Boolean value.
The No attribute, if specified, is not a valid 32-bits integer number.
The UserType attribute, if specified, has not a valid value.
The UserTypeNo attribute, if specified, is not a valid 32-bits integer number.
<NoEMailAddress> The user has no e-mail address.
<NoUserFound> No user has been found with the parameters specified.
<ParameterMissing> You must specify one whole group of attributes.
 Security

This request is public and can be used by any user.
